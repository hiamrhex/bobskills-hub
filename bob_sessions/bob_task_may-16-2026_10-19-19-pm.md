123 | ## Best Practices for TypeScript Conversion
124 | 
125 | ### 1. Start with Strict Mode
126 | Enable TypeScript's strict mode from the beginning to catch potential issues early and ensure maximum type safety.
127 | 
128 | ### 2. Prefer Interfaces Over Type Aliases for Objects
129 | Use interfaces for object shapes as they're more extensible and provide better error messages.
130 | 
131 | ```typescript
132 | // Preferred
133 | interface User {
134 |   id: number;
135 |   name: string;
136 |   email: string;
137 | }
138 | 
139 | // Use type aliases for unions, intersections, or primitives
140 | type Status = 'active' | 'inactive' | 'pending';
141 | ```
142 | 
143 | ### 3. Use Explicit Return Types
144 | Always specify return types for functions to catch errors and improve code documentation.
145 | 
146 | ```typescript
147 | function calculateTotal(items: number[]): number {
148 |   return items.reduce((sum, item) => sum + item, 0);
149 | }
150 | ```
151 | 
152 | ### 4. Leverage Type Inference
153 | Don't over-annotate; let TypeScript infer types when it's obvious.
154 | 
155 | ```typescript
156 | // Good - type is inferred
157 | const count = 5;
158 | 
159 | // Unnecessary
160 | const count: number = 5;
161 | ```
162 | 
163 | ### 5. Use Readonly for Immutable Data
164 | Mark properties as readonly when they shouldn't be modified after initialization.
165 | 
166 | ```typescript
167 | interface Config {
168 |   readonly apiUrl: string;
169 |   readonly timeout: number;
170 | }
171 | ```
172 | 
173 | ### 6. Avoid `any` Type
174 | Replace `any` with proper types, `unknown`, or generics to maintain type safety.
175 | 
176 | ```typescript
177 | // Bad
178 | function process(data: any): any {
179 |   return data;
180 | }
181 | 
182 | // Good
183 | function process<T>(data: T): T {
184 |   return data;
185 | }
186 | ```
187 | 
188 | ### 7. Use Union Types for Multiple Possibilities
189 | Leverage union types instead of loose typing.
190 | 
191 | ```typescript
192 | type Result = Success | Error;
193 | type Status = 'loading' | 'success' | 'error';
194 | ```
195 | 
196 | ### 8. Create Utility Types for Common Patterns
197 | Define reusable utility types for your domain.
198 | 
199 | ```typescript
200 | type Nullable<T> = T | null;
201 | type Optional<T> = T | undefined;
202 | type ApiResponse<T> = {
203 |   data: T;
204 |   status: number;
205 |   message: string;
206 | };
207 | ```

209 | ## Common Pitfalls to Avoid
210 | 
211 | ### 1. Over-using Type Assertions
212 | Type assertions (`as Type`) bypass type checking and should be used sparingly.
213 | 
214 | ```typescript
215 | // Avoid
216 | const user = data as User;
217 | 
218 | // Better - validate the data
219 | function isUser(data: unknown): data is User {
220 |   return typeof data === 'object' && data !== null && 'id' in data;
221 | }
222 | ```
223 | 
224 | ### 2. Ignoring Null/Undefined
225 | Always handle potential null or undefined values properly.
226 | 
227 | ```typescript
228 | // Bad
229 | function getName(user: User): string {
230 |   return user.name; // What if user is null?
231 | }
232 | 
233 | // Good
234 | function getName(user: User | null): string {
235 |   return user?.name ?? 'Unknown';
236 | }
237 | ```
238 | 
239 | ### 3. Not Using Discriminated Unions
240 | For complex state, use discriminated unions instead of optional properties.
241 | 
242 | ```typescript
243 | // Bad
244 | interface State {
245 |   loading?: boolean;
246 |   data?: Data;
247 |   error?: Error;
248 | }
249 | 
250 | // Good
251 | type State = 
252 |   | { status: 'loading' }
253 |   | { status: 'success'; data: Data }
254 |   | { status: 'error'; error: Error };
255 | ```
256 | 
257 | ### 4. Mixing Interfaces and Implementation
258 | Keep type definitions separate from implementation logic.
259 | 
260 | ```typescript
261 | // Bad - mixing concerns
262 | interface User {
263 |   id: number;
264 |   getName(): string; // implementation detail
265 | }
266 | 
267 | // Good - separate interface and implementation
268 | interface User {
269 |   id: number;
270 |   name: string;
271 | }
272 | 
273 | class UserImpl implements User {
274 |   constructor(public id: number, public name: string) {}
275 |   getName(): string {
276 |     return this.name;
277 |   }
278 | }
279 | ```
280 | 
281 | ### 5. Not Leveraging Type Guards
282 | Use type guards for runtime type checking instead of type assertions.
283 | 
284 | ```typescript
285 | function processValue(value: string | number) {
286 |   if (typeof value === 'string') {
287 |     // TypeScript knows value is string here
288 |     return value.toUpperCase();
289 |   }
290 |   // TypeScript knows value is number here
291 |   return value.toFixed(2);
292 | }
293 | ```
294 | 
295 | ### 6. Forgetting to Update Tests
296 | When converting to TypeScript, update test files and add proper typing to test utilities.
297 | 
298 | ### 7. Not Using Generics When Appropriate
299 | Generics provide type safety for reusable code without sacrificing flexibility.
300 | 
301 | ```typescript
302 | // Bad - loses type information
303 | function getFirst(arr: any[]): any {
304 |   return arr[0];
305 | }
306 | 
307 | // Good - preserves type information
308 | function getFirst<T>(arr: T[]): T | undefined {
309 |   return arr[0];
310 | }
311 | ```
312 | 
313 | ### 8. Incomplete Type Definitions
314 | Ensure all properties and methods are properly typed, including optional ones.
315 | 
316 | ```typescript
317 | // Incomplete
318 | interface User {
319 |   id: number;
320 |   name: string;
321 | }
322 | 
323 | // Complete
324 | interface User {
325 |   id: number;
326 |   name: string;
327 |   email?: string; // optional property
328 |   createdAt: Date;
329 |   updatedAt: Date;
330 | }
331 | ```
